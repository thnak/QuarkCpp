# ADR-028: Lazy Activation-on-First-Message + Idle-Timeout Deactivate Eviction

## Status

Accepted

## Question

Design and prove the actor activation lifecycle seam that is currently unimplemented:

1. **IdleTimeout<Ms>-driven Deactivate eviction** (011/ADR-008 specify a wheel-driven arm-on-drain-Empty, fire-and-evict-after-seq_cst-close-out mechanism that is sim-proven but absent from `activation.hpp`/`engine.hpp`/`timer_wheel.hpp`).
2. **Lazy activation on first message** for an `ActorId` with no live `Schedulable` (`Engine::resolve()` today is a pure `by_id_` lookup returning `nullptr` — marked in code as "a seam to 010/002 lazy activation").

These must be designed together: evicting an activation without a reactivation path makes it permanently unreachable. The design had to pin down concurrency for (a) N producer threads racing first-touch of the same never-active/evicted `ActorId`, (b) a Deactivate firing concurrently with an inbound message, (c) ADR-021 (Eager-Everywhere) compliance — Node/Shard resources resolved once at `Engine::build()`, never re-resolved by reactivation, (d) hot-path cost of the resolve check on every message, and (e) failure semantics when a 012-Persistence reload fails during reactivation.

## Designs Considered

1. **Pre-sized Open-Addressed Slot Table + CAS Lifecycle State Machine** — a permanent, per-ActorId, open-addressed `ActivationSlot` table sized at `Engine::build()`; first-touch and reactivation arbitrated by a CAS on an `ActivationState` enum; parking via `atomic::wait`/`notify`; a per-caller `ActivationReadGuard` (QSBR-style beacon) added post-hoc to close a heap-use-after-free found during cross-examination.
2. **Broker-Actor Serialization** — one built-in per-shard `ActivationBroker` actor that reuses Quark's own single-executor guarantee to arbitrate first-touch/reactivation instead of inventing new CAS/RCU machinery; a new sealed `Dormant` ExecState reuses the existing seq_cst mailbox-close-out Dekker fence verbatim, parameterized to a new target; a permanent, per-shard, single-writer (broker-only) lock-free id-table replaces `Engine::by_id_` for addressing.
3. **Generation-Tagged Identity Word + Epoch-Deferred Reclamation** — one 8-byte `atomic<PackedIdentity>` word per `ActorId` (tag + generation + pointer), Fraser-style EBR for reclamation, first-touch/reactivation via CAS on the packed word, eviction via a new `Retired` ExecState folded into the existing close-out sequence.

## Decision: **Broker-Actor Serialization** wins.

**Generation-Tagged Identity Word is disqualified by the safety gate.** Three independent safety/correctness claims (S1, C1, C3) were proven **WRONG** with executed evidence, and all three trace to the *same* unfixed root defect: an unsynchronized gap between "the park-mailbox/mailbox drain confirms empty" and "the state-word publish/retire CAS commits," which the design's own cross-examination round claimed to have closed and did not. This is not a benchmark-noise or documentation quibble — it is the literal ground-rule invariant "a message is never lost" failing under proof, 3 separate times, in 3 separate code paths (first-touch construction, eviction close-out, and reconstruction-failure draining). No cheap fix was demonstrated; the rebuttal's proposed fix for exactly this class of bug (a re-check after the CAS) was not present in the code that was actually built and proven, and the same unsynchronized-gap shape recurring three times signals this is structural to the design's "two independently-synchronized primitives, no linearization between them" approach, not an isolated slip. Per ranking rule (1), this disqualifies it outright.

**Pre-sized Slot Table survives the safety gate (no safety/correct claim was proven WRONG) but loses on ranking rule (3), decisively.** Its own core "steady-state hot path pays nothing beyond a predicted branch" claim (F1) was proven **WRONG** — not marginally: +47% to +167% p50/p99 latency regression versus baseline on both compilers, with GCC compiling the required per-call QSBR beacon into a `lock orq` instruction sitting directly on the hot tell/ask path. That is a direct violation of this project's ground rule "Zero-cost hot path... no dynamic resource resolution while a message is processed," proven by objdump, not argued. The design also required a genuinely undocumented fix (a deadlock in its own QSBR beacon scheme, found and patched only during proving) to reach its otherwise-clean safety verdicts — a sign the design's true complexity/fragility was underestimated even after two rounds of cross-examination.

**Broker-Actor Serialization has zero disproven safety/correctness claims** — every one of S1, S2, S3, S4, C1, C2, C3 was proven CORRECT with executed evidence, including two adversarial bugs (a `Shard::owned_activations_` UAF under concurrent broker workers, and a double-dispatch race in `handle_wake`'s erase-ordering) that were *found and fixed during the proving run itself*, then re-verified clean at full spec scale. Its hot-path claim (F1) is also proven CORRECT with the best measured numbers of any surviving design: throughput within noise of the current baseline (−6.07%…−2.11% across compiler/budget combinations, plausibly attributable to machine noise given a co-resident process was observed consuming 150–180% CPU during the run) and an objdump-verified single new predicted-branch instruction on the cold leg, zero new atomic loads, zero lock/cmpxchg/xadd anywhere on the steady-state path. Its one weak spot, F2 (cold-activation latency), is **INCONCLUSIVE, not disproven** — the 13–35x end-to-end number is an artifact of this prototype modeling the broker as an OS-thread woken via `condvar`/mutex rather than the design's own intended engine-scheduled actor lane; the broker-internal-only number (~2.04x median) is much closer to the claimed bound. Per the ranking rules, INCONCLUSIVE carries no weight against the design, and it does not carry a fatal safety violation, so Broker-Actor Serialization is the winner both on the safety gate and on measured hot-path numbers.

## Residual Risks

1. **F2 (cold-activation latency) is unresolved.** The prototype's blocking-condvar broker-thread model cannot cleanly confirm or falsify the ≤2x claim; the real implementation must route the broker through the engine's own low-latency scheduler wakeup, not a blocking OS primitive, before this number can be trusted. **This is the single tying-breaking experiment to run next**: re-implement the broker as a genuine engine-scheduled actor (not a std::thread + condvar stand-in) and re-measure p50/p99 first-touch latency against `spawn<A>()` end-to-end.
2. **Broker convoy risk.** Even with `MaxConcurrency<K>` reentrancy, a burst of first-touches/reactivations exceeding K on one shard, or one pathologically slow `EventSourced` replay, queues later requests behind it. Needs a 009 counter (`broker_queue_depth` / `broker_stall_ms`) to make this observable in production, not just bounded in theory.
3. **Unbounded id-table growth.** The per-shard id-table is permanent and never shrinks; a workload with huge-cardinality, mostly-one-shot `ActorId`s grows it unboundedly. No compaction/GC seam is built. Needs sizing guidance and a 009 counter at minimum; a future compaction scheme is out of scope for this ADR.
4. **S4's relaxed-ordering positive control did not visibly tear on this x86-TSO box even under the deliberately-broken build** (it did fire under TSan, which reasons about the C++ memory model, not observed hardware behavior). ARM64 promotion requires this gate to be re-run on that architecture before any port, consistent with every other Dekker-style fence in this codebase.
5. **Broker-internal supervision policy for a genuine bug (not a StateStore error) mid-reload is still unspecified** — if `construct_and_wire()` itself faults, whether the broker's own supervision unwinds just that one `activating_` entry (correct) or, via a naive policy, restarts the whole broker and drops every *other* shard's in-flight activation (wrong) needs a dedicated SupervisionPolicy, not inherited defaults. Flagged, not resolved, by this ADR.
6. **The two bugs found-and-fixed during proving** (unsynchronized `Shard::owned_activations_` under K≥2 broker workers; `handle_wake`'s premature `activating_` erase before the Dormant→Idle transition) must be carried into the actual implementation as explicit, permanently-regression-tested fixes — they are not incidental prototype artifacts, they are exactly the kind of subtle ordering bug this ADR exists to close, and the winning design's *proven* form differs from its as-originally-cross-examined form in these two specific ways.
7. **TSan does not model standalone `std::atomic_thread_fence`.** S2's [Deactivate,M] guarantee was proven via the repo's own established counting-based Dekker-litmus methodology (matching `tests/stream_close_out_dekker_test.cpp`), not via TSan directly. This is consistent with existing project practice, not a new gap, but should stay documented as a proof-methodology caveat wherever this fence is cited.

## Spec Recommendations

- **`011-Timers-and-Scheduled-Work.md`**: Specify the winning `Dormant` ExecState mechanism precisely: the Deactivate wheel-fire is dispatched as an ordinary internal descriptor (a reserved flag bit on the already-loaded gen_state word, not a new memory load) that a `drain_step` dispatch recognizes and converts into a private, non-atomic `retire_requested_` flag — **critically, the Deactivate handler itself must never touch ExecState or the identity table directly** (this was the exact defect that made the losing Slot-Table design's original close-out integration unsafe). The actual retire attempt happens only inside the *existing* `close_out()` call site on `DrainedEmpty`, adding exactly one new terminal outcome (`Retired`/`Dormant`) to the existing release→fence→probe→reacquire sequence, so the proven [Deactivate,M]-abort-eviction Dekker fence is reused byte-for-byte, never re-derived. Document the busy-edge generation bump that invalidates stale Deactivate tokens on wake.
- **`004-Resources.md`**: Add an explicit clause that lazy (re)activation NEVER re-resolves Node- or Shard-scoped resources — the broker's `construct_and_wire` reads the already-eager, build()-time `ResourceScope` table exactly as `spawn<A>()` does today; only Activation-scoped `Cached<>`/`PerMessage<>` members are freshly wired per generation via a distinct `wire_activation_scoped()` step. State plainly that "no dynamic resolution while draining" extends to the reactivation path, not just steady-state dispatch.
- **`012-Persistence.md`**: Specify two distinct reload-failure paths with two distinct outcomes: (a) first-touch reload failure fails the triggering message plus every message buffered behind it via the 007 boundary (tell → dead-letter, ask → `unexpected(resource_error)`), and the `ActorId` remains cold/retry-eligible, never poisoned; (b) reactivation-of-Dormant reload failure runs a dedicated `fail_drain_via_007` pass — the same Dekker drain/fence/probe shape as ordinary close-out, but failing instead of dispatching — against everything resident in the mailbox at failure time, and the activation stays Dormant (not Idle) afterward; a later, unrelated message re-triggers an independent Wake/retry. Document the accepted residual risk: no backoff/circuit-breaker exists yet against a permanently-broken store being hammered by every subsequent message.
- **`013-Configuration.md`**: Add `broker_concurrency` (the per-shard `MaxConcurrency<K>` reentrancy bound on the `ActivationBroker`) as a documented BuildOnly-tunable knob, plus a 009 alarm counter for broker queue depth/stall, per residual risk #2.
- **`001-Actor-Execution-Model.md`**: Add `Dormant` to the `ExecState` enum next to `Idle`/`Scheduled`/`Running`/`Parked`; document its semantics ("no live actor object right now — needs the broker before this ActorId can run again") and that `notify_enqueued()`'s new `WentDormant` outcome is the ONLY new branch on the hot `post()` path (one predicted compare against the CAS's failure-observed value), firing a lightweight `Wake{id}` to the shard's broker rather than moving the message.
- **`002-Scheduler.md`**: Note that the per-shard `ActivationBroker` is itself an ordinary scheduled actor — Redirect/Wake messages are scheduled exactly like any user message, with no new scheduling primitive. Re-emphasize "scheduler schedules activations, not messages" applies unchanged to the broker.
- **`decisions/ADR-008-engine-actor-configuration-and-activation-lifecycle-policy.md`**: Replace the "seam to 010/002 lazy activation" placeholder with a citation to this ADR and the concrete winning mechanism (broker-actor serialization, permanent per-shard id-table, `Dormant` ExecState). Record the two bugs found-and-fixed during proving (owned_activations_ needing its own synchronization under K≥2 broker workers; handle_wake's erase-after-not-before Dormant→Idle transition) as permanent regression-test line items.
- **`decisions/ADR-021-node-shard-resource-resolution-ordering.md`**: Close the residual risk this ADR itself flagged ("the extension to a true lazy path is sound but argued, not executed... must re-run C1 the moment the lazy-activation seam lands"). Mark it resolved: C1 was re-run against the actual lazy-activation broker path and proven CORRECT (Node-scope factory invocation count stays at 1 across 1000 evict/reactivate cycles). Cite ADR-028.
